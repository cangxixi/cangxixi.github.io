Two disadvantages of the ESRI File Geodatabase¶

Xuezhi Cang Email: xuezhicang@gmail.com Website: https://xuezhicang.com/

Two disadvantages: 1) the FGBD does not fully support the multiprocessing library; 2) I/O ofFGDBis slow if you have a big set of data.

A File GeoDataBase (FGDB) is a collection of files in a folder on disk that can store, query, and manage both spatial and nonspatial data. (https://desktop.arcgis.com/en/arcmap/10.3/manage-data/administer-file-gdbs/file-geodatabases.htm#:~:text=A%20file%20geodatabase%20is%20a,Feature%20class)

Using feature class and raster dataset in the FGDB, instead of using shapefile and grid data in the ordinary oler, is recomanded by ESRI. The main reasons are: 1) there is a 2 GB size limit for any shapefile component file; and 2) the name length of grid file must be smaller than 13 characters.

However, the GDB is not always better than the ordinoray folder when we deal with a large amount of files. The example is shown as below.

Our task¶

We have a global Martian flow accumulation raster data, which is generated from the gloabl Martian DEM data, and a global Martian watersheds vector data consisting of a lot of polygons. Our task is to 1) split the watershed polygons to single files based on their ID, 2) extract the flow accumulation raster data to single raster files by using each watershed, and 3) export the watershed boundary to CSV file, whose watershed area is labeled as 1 and outlet cell is labeled as 2.

Original Strategy¶

The SearchCursor (https://pro.arcgis.com/en/pro-app/latest/arcpy/data-access/searchcursor-class.htm) is used to select and paste each small watershed polygon from the global watersheds data. Then, every small flow accumulation area is extracted and stored in the GDB. At last, every CSV is generated from the corresponding flow accumulation file.

Since the watershed feature class has a lot of watersheds, this program is extremely SLOW. It needs more than five hours to process 2,000 watershed polygons. The total number of watersheds are more than 15,000, so this speed is NOT ACCEPTDABLE.

Improved Strategy¶

Parallel Computing¶

Since the tasks of function in the for-loop are similar and do not depend on each other. The parallel computing should be a good idea to accelerate our program's speed!

A sample Python parallel computing code (from https://docs.python.org/3/library/multiprocessing.html) is shownn as below:

Multiprocessing¶

from multiprocessing import Pool

def f(x):
    return x*x

if __name__ == '__main__':
    with Pool(5) as p:
        print(p.map(f, [1, 2, 3]))

Problem 1¶

Following the structure of the example, I build my multiprocessing version program. Unfortunately, it does not work. The reason is that we cannot use pool function to write files to the FGDB. The explanation is shown as below:

https://www.e-education.psu.edu/geog489/node/2261

Solution for Problem 1¶

The shapefile and grid raster are writen to the ordinary folder by using the pool function. It greatly accelerates the computing speed.

Problem 2¶

In my program, the small flow accumulation area is extracted by the extract-by-mask funtion (see: https://pro.arcgis.com/en/pro-app/latest/tool-reference/spatial-analyst/extract-by-mask.htm). When the extract-by-mask funtion is applied to the global flow accumulation raster file, it may report error (code: 999999). This error occurrs randomly, but frequentlly. The possible reason is that the extract-by-mask function needs to lock the global flow accumulation file to prevent other core's visit, and then causes the crash. (I hope that I make a good guess! If you know the real reason, please let me know)

Solution for Problem 2¶

To solve this problem, I have to use a for-loop to extract the small flow accumulation area by using every single watershed file and use the pool function to split the watershed polygons and export the CSV files (I know it is awkward, but it works at least.)

After solving the two problems, my code are shown as bwlow:

I/O speed¶

As I mentioned in the Problem 2 section, I use a for-loop to extract the flow accumulation area. If the global flow accumulation data is in the FGDB, the program takes a lot of time. If the flow accumulation is converted to TIFF file and stored in an ordinary folder, the processing speed will be improved greatly. I think that the reason is the I/O for the GDB is much much slower than the I/O for the ordinary folder.

The code for the comparsion is shown as below:

start_time = time.time()
# extract the facc raster by mask
for i in ws_feature_IDs: 
    # extract the csv file and save it to folder 
    # flow_acc_data is the global flow accumulation data
    flow_acc_ExtractByMask = ExtractByMask(Raster(flow_acc_data), i[4] + "ws_"+str(i[0]) + ".shp")
    flow_acc_ExtractByMask.save(i[5] +"fa_"+ str(i[0]))
end_time = time.time()
print("used time(in seconds): " + str(end_time - start_time))

To simplify the process, I only extract 10 flow accumulation files from the global flow accumulation file.

If the global flow accumulation is stored in the GDB, it takes 20.77s for 10 watersheds.

If the global flow accumulation is stored in the ordinary folder, it takes 5.47s for 10 watersheds.

Considering that we will process more than 15,000 watershed polygons, we need to wait a lot of time if the GDB is used here.

Summary¶

Although the GDB has a lot of advantages over the ordinary folder, we should use the ordinary folder to store our files (shapefiles, TIFF, etc.) if we want to use multiprocessing library to speed up our program.
